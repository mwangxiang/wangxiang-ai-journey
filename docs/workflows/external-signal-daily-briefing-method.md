# Workflow：如何做可复刻的“外部发现日报”，并自动产出 Markdown 文档

这个 workflow 解决的不是“怎么写一份好看的日报模板”，而是更实际的问题：

> 当用户确定想看的领域后，如何稳定地搜集外部信息、提炼信号、形成判断，并自动交付一份方便阅读、方便转发、方便归档的 Markdown 文档。

这套方法适合公开分享，因为它：
- 不依赖个人背景
- 不依赖特定聊天上下文
- 能被别人直接照着复刻
- 最终输出是 `.md` 文档，方便查看、修改、存档和二次分发

---

## 一、适用场景

适合下面这些场景：

- 想每天或定期了解某个领域的最新动态
- 不想自己刷很多平台
- 不想看平台流水账
- 想快速知道“今天真正值得关注的是什么”
- 想让 AI 在搜集后直接交付一份文档，而不是只在聊天里零散回复

常见领域示例：
- AI / agent / workflow
- GitHub 开源项目动态
- 某个行业的产品趋势
- 某个技术方向的外部信号
- 某个垂直领域的舆情和新工具

---

## 二、最终交付目标

这套 workflow 的最终目标不是“回答一段话”，而是：

**在确认用户关注的领域后，自动生成一份 Markdown 报告文档。**

建议交付形式：
- 文件格式：`.md`
- 内容语言：按用户偏好
- 文件内容可直接阅读，也可继续转为 PDF、发 GitHub、发飞书、发 QQ 文件

### 推荐文件名格式

```text
领域名-daily-briefing-YYYY-MM-DD.md
```

例如：

```text
ai-agent-daily-briefing-2026-03-17.md
open-source-ai-daily-briefing-2026-03-17.md
中国AI产品观察-daily-briefing-2026-03-17.md
```

如果用户不需要“日报”频率，也可以改为：

```text
领域名-signal-report-YYYY-MM-DD.md
```

## 三、核心原则

### 1. 先确认领域，再开始搜
不要一上来就全网乱搜。

先确认用户真正想看的是什么，例如：
- 是 AI 行业，还是某个子方向？
- 是偏技术、偏产品、偏商业，还是偏舆情？
- 是国内信息优先，还是全球信息优先？
- 是想看“今天发生了什么”，还是想看“最近趋势”？

只有先把领域收窄，后面的搜集和判断才有质量。

### 2. 先搜集，再谈模板
模板不是第一优先级。

第一优先级是：
**先把值得汇报的内容抓出来。**

如果内容本身没提纯出来，再漂亮的模板也只是空壳。

### 3. 不按平台堆内容，要按“信号类型”提炼
坏写法通常是：
- Twitter 有什么
- GitHub 有什么
- 小红书有什么
- B站有什么

这样很容易变成信息垃圾堆。

更有效的方式是按“信号类型”整理：
- 爆发信号
- 转向信号
- 工具信号
- 情绪信号
- 机会信号

也就是说：
**不是问“哪里有内容”，而是问“今天出现了什么值得注意的变化”。**

### 4. 每条都要回答“为什么值得看”
只复述事实不够。

每条信息都应补上：
- 为什么值得关注
- 它意味着什么
- 它是偶发噪音，还是趋势的一部分
- 对后续判断有什么帮助

### 5. 最终必须落到 Markdown 文档
不要只停留在聊天消息里。

因为聊天消息：
- 不方便归档
- 不方便转发
- 不方便后续加工
- 不方便沉淀成长期资产

而 Markdown 文档：
- 轻量
- 通用
- 可版本管理
- 可直接进 GitHub
- 可再转为其他格式

### 6. 工具失效时，要立刻换路线
真实环境里，搜索 API、网页抓取、浏览器工具都可能失效。

所以要把“换路线”视为标准能力，而不是意外。

例如：
- 搜索 API 挂了 → 改用 GitHub API / 指定站点抓取 / 媒体源直抓
- 浏览器工具失效 → 改抓网页正文 / API / raw 页面
- 搜索结果太散 → 改成盯固定对象和固定信号源

## 四、推荐的信号分类法

### 1. 爆发信号
短时间内突然被很多人提到、转发、测试的东西。

例如：
- 某个新产品突然刷屏
- 某个模型突然被集体测评
- 某个概念突然成为讨论焦点

价值在于：
**它告诉你“大家突然在看什么”。**

### 2. 转向信号
不是最热，但能说明风向开始变化。

例如：
- 讨论重点从“模型能力”转到“成本/工作流”
- 从“酷炫 demo”转到“持续可用性”
- 从“单体 agent”转到“工作流 / orchestration / macro layer”

价值在于：
**它告诉你“行业正在往哪偏”。**

### 3. 工具信号
最近冒出来、值得试或值得跟踪的工具、仓库、产品。

例如：
- 新 GitHub 项目
- 新插件
- 新自动化方案
- 某个生态里的新壳层 / 新中间层

价值在于：
**它告诉你“有哪些具体对象值得点进去看”。**

### 4. 情绪信号
大家最近在兴奋什么、吐槽什么、担心什么。

例如：
- 大家开始担心 agent 安全边界
- 大家对安装门槛很不耐烦
- 大家不再只看能力，而开始问“能不能长期跑”

这个类型很重要，因为情绪经常先于数据表达趋势。

### 5. 机会信号
哪些变化意味着新的产品机会、内容机会、交付机会。

例如：
- 复杂能力需要更简单入口
- 原始能力需要更好的产品包装
- 技术项目需要“服务化交付”才能真正被大众使用

价值在于：
**它帮助把“观察”转成“判断”。**

## 五、标准执行步骤

### Step 1：确认用户关注的领域
先问清楚：
- 领域是什么
- 关注的是技术、产品、商业，还是综合
- 关注范围是国内、国外，还是都看
- 输出要偏“今天动态”还是偏“趋势观察”

这一步一定要先做，因为后面的所有搜集都依赖这个边界。

### Step 2：确定观察范围与信息源
常见信息源包括：
- 搜索 API / 搜索引擎
- GitHub 仓库、Topics、Release、Issue
- 媒体报道
- 官方博客 / 更新日志
- 社交平台公开内容
- 固定观察对象（某几个项目、公司、产品）

建议先广扫一轮，拿到素材池。

### Step 3：筛选并归类
拿到原始素材后，不要直接原样汇报。

先按信号分类：
- 这是爆发信号吗？
- 这是转向信号吗？
- 这是工具信号吗？
- 这是情绪信号吗？
- 这是机会信号吗？

### Step 4：压缩成最值得看的 3～5 条
不要贪多。

大多数用户真正需要的是：
- 今天最值得看的几条
- 每条为什么值得看
- 每条背后意味着什么

这一步是日报质量的核心。

### Step 5：提炼趋势总结
在条目之外，再加一段总结：
- 今天这些信号背后有没有共同方向
- 它们是分散噪音，还是同一个大趋势的不同侧面

### Step 6：整理成 Markdown 文档
当条目和总结都完成后，必须落到 `.md` 文件。

建议文档包含：
- 标题
- 日期
- 关注领域
- 今日重点 3～5 条
- 趋势总结
- 值得继续跟踪的对象
- 下一步建议
- 原始来源链接（可选）

### Step 7：把文档作为正式交付物发送
最终不是只回复“我搜完了”，而是：
- 生成 `.md` 文件
- 直接发文件给用户，或写入指定项目 / 仓库 / 文档系统

## 六、推荐的 Markdown 输出结构

下面是建议的 `.md` 文档结构：

```md
# {领域名称} 外部发现快报 - YYYY-MM-DD

## 今日关注范围
- 领域：
- 地域范围：
- 侧重点：

## 今日最值得看的 3～5 条

### 1. 标题
- 发生了什么
- 为什么值得看
- 这意味着什么

### 2. 标题
- 发生了什么
- 为什么值得看
- 这意味着什么

## 今日趋势总结
- 趋势 1
- 趋势 2
- 趋势 3

## 值得继续跟踪的对象
- repo / 产品 / 公司 / 关键词 / 媒体源

## 下一步建议
- 明天继续看什么
- 哪些值得深挖
- 哪些可以忽略

## 可选：原始来源
- 链接 1
- 链接 2
- 链接 3
```

## 七、最小可复刻流程

如果要让别人一看就能复刻，最小流程可以压成下面这 7 步：

1. 先确认用户想看的领域
2. 确定观察范围和优先信息源
3. 快速扫一轮，拿到素材池
4. 按信号类型筛选归类
5. 压缩成最值得看的 3～5 条
6. 补趋势总结和后续建议
7. 输出为 `.md` 文档并发送

只要照着这 7 步走，就能稳定复刻一版可用的“外部发现日报”。

---

## 八、复刻时最常见的错误

### 错误 1：一上来就搜太大范围
结果是信息太散，最后无法判断什么最重要。

### 错误 2：按平台堆内容
结果是看起来很多，实际上没有信号提纯。

### 错误 3：只报事实，不做判断
结果是用户知道“发生了什么”，但不知道“为什么值得看”。

### 错误 4：输出只停留在聊天消息
结果是内容难以归档，也不方便继续复用。

### 错误 5：没有先确认用户领域
结果是搜出来的内容和用户真正关心的方向不一致。

---

## 九、为什么这套方法适合写进项目

因为它不是一次性的聊天技巧，而是一个可以长期复用的工作流：

- 可以跨领域使用
- 可以跨平台使用
- 可以适配不同信息源
- 可以稳定产出标准化的 `.md` 文档
- 可以沉淀进 GitHub 项目、文档库或知识库

它体现的不是“AI 会聊天”，而是：
**AI 能参与真实的信息发现、提纯、判断和文档交付流程。**

---

## 十、总结

一句话概括这套 workflow：

> 在确认用户关注领域后，先搜集外部信息，再按信号提纯，最后自动交付一份 Markdown 快报文档。

这套方法的关键不在“日报”两个字，而在：
- 有边界
- 有筛选
- 有判断
- 有正式交付物

只有这样，别人看完才能真正复刻，用户拿到后也真正方便使用。
