# Workflow：如何持续维护一个公开 GitHub 项目

这个工作流不是针对大团队，而是针对像 `WangXiang AI Journey` 这种由个人主导、持续公开构建的项目。

## 核心目标

维护公开项目时，最容易出问题的不是“不会写”，而是：
- 不知道先写什么
- 内容越来越散
- 只顾眼前更新，没有长期结构
- 更新几次后就断掉

所以维护工作流的核心，不是加速，而是：
**降低持续维护的摩擦。**

## 一、先把仓库骨架搭好
在开始大量写内容之前，先准备好：
- README
- ROADMAP
- CONTRIBUTING
- CHANGELOG
- SECURITY
- 内容目录结构
- 脱敏规范

这样后续新增内容时，知道放哪里，也知道按什么原则写。

## 二、把内容分成几类
建议至少拆成：

### 1. Learning Log
记录今天推进了什么，偏时间线。

### 2. Case Study
记录完整案例，偏复盘。

### 3. Workflow
记录可复用流程，偏方法。

### 4. Guide
记录新手入口，偏教学。

这四类内容混在一起写，仓库很快就会乱；
分开以后，结构会稳定很多。

## 三、每次更新尽量带着“为什么”
不要只写：
- 改了什么
- 加了什么

更要写：
- 为什么这样改
- 为什么不用别的方案
- 这次调整解决了什么问题

因为读者最需要的，往往不是结果，而是判断过程。

## 四、更新时同步写变更记录
每做一批明显变更，就同步更新 CHANGELOG。

这样做的好处：
- 方便回顾
- 方便对外展示推进节奏
- 方便后面提炼简历项目描述

## 五、对外公开前先做脱敏检查
即使是记录项目过程，也不要直接把原始内容原封不动贴出去。

发布前至少检查：
- 有没有 token / key
- 有没有私人身份信息
- 有没有敏感基础设施信息
- 有没有不适合公开的原始聊天内容

## 六、维护节奏不要太理想化
个人项目最容易失败的原因之一，就是一开始把节奏定得太满。

更实际的做法是：
- 有推进时就记录
- 一次更新一小批
- 先保证不断，再追求精致

## 七、定期回头整理首页
当内容开始多起来后，README 必须跟着更新。

否则会出现一个典型问题：
仓库里有内容，但新来的人一眼看不出来最值得看的是什么。

所以首页应该定期补：
- 精选内容入口
- 当前阶段说明
- 值得先看的 3~5 篇内容

## 这个工作流适合什么项目

适合：
- build in public 项目
- 个人 AI 助手实践项目
- 持续公开学习项目
- 个人作品集型仓库

不太适合：
- 一次性代码作业
- 纯代码库、几乎不写文档的项目

## 总结

持续维护公开 GitHub 项目，不靠热血，靠的是：
- 结构先行
- 分类清楚
- 更新有因
- 脱敏严格
- 节奏可持续

这也是当前这个项目正在实践的维护方式。
