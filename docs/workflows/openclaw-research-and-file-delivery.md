# Workflow：如何用 OpenClaw 快速、准确地查资料并把结果发送给用户

这个 workflow 来自一次真实任务：
用户想找 `S.H.I.T Journal` 里一篇具体论文——**《地府货币膨胀：东亚父母该烧多少钱才能保证孩子不会乱花》**，并且希望最终能**直接收到文件**，而不是只收到一个可能打不开的链接。

这个过程很适合沉淀成标准方法，因为它同时涉及：
- 模糊目标定位
- 网站结构识别
- 浏览器失效时的替代方案
- API 逆向式查找
- 文件下载与本地转发
- 面向用户的结果交付

---

## 一、适用场景

适合下面这类任务：

- 用户只记得一个模糊标题或关键词
- 用户要找的是某个网站里的具体文章/论文/文件
- 普通搜索结果噪音很大
- 用户最终想要的是“把文件发给我”而不是“给我一个链接自己找”
- 浏览器工具、搜索 API、网页抓取中途可能失效

---

## 二、核心原则

### 1. 不要一开始就死盯搜索引擎
当目标位于**特定网站内部**时，外部搜索往往噪音太高。

更有效的顺序通常是：
1. 先确认官网
2. 再确认前端结构
3. 再找 API / 数据源
4. 最后再精确检索目标内容

### 2. 浏览器不是唯一入口
如果浏览器工具不可用，不要卡住。

可替代路线包括：
- 抓首页 HTML
- 抓前端 JS 包
- 找 API 域名
- 看 OpenAPI / Swagger 文档
- 直接请求 JSON 接口

### 3. 面向交付，而不是停在“我找到了”
用户很多时候不是想听过程，而是想要：
- 文件本身
- 可直接打开的内容
- 一段清晰结论

所以最终目标应当是：
**把东西送到用户手里。**

---

## 三、标准执行步骤

### Step 1：先定位站点身份
如果用户只给了一个模糊名字，例如：
- `S.H.I.T`

先确认它到底是什么：
- 官网是什么
- 是期刊、博客、论坛还是资源库
- 有没有固定域名

本次任务里，最终确认官网为：
- `https://shitjournal.org/`

---

### Step 2：如果常规搜索不稳定，改查站点前端
当搜索接口不稳定、浏览器工具失效时，可以直接：
- 抓首页 HTML
- 抓前端 JS bundle
- 从 JS 中提取：
  - 文案
  - 路由
  - API 域名
  - 数据结构

本次任务中，从前端代码里成功提取到：
- API 域名：`https://api.shitjournal.org`
- 站点栏目与文案
- 文章展示结构
- 搜索和文章详情相关路径线索

---

### Step 3：优先找公开 API 文档
如果目标站点暴露了 OpenAPI / Swagger / Redoc，这是最省时间的突破口。

本次任务中成功发现：
- `https://api.shitjournal.org/openapi.json`
- `https://api.shitjournal.org/docs`
- `https://api.shitjournal.org/redoc`

这一步的价值非常高，因为它能直接告诉你：
- 真正可用的接口路径
- 查询参数
- 单篇详情接口
- 搜索接口

---

### Step 4：先拉列表，再用搜索接口精确命中
推荐顺序：

#### 4.1 先看文章总量和分页
例如：
- `/api/articles?limit=20&page=1`

先知道库里大概有多少内容，避免瞎猜。

本次任务里确认：
- 总数：31 篇
- 共 2 页

#### 4.2 再用官方搜索接口做精确检索
本次任务中，真正命中目标的是：
- `/api/search/article?q=地府货币膨胀`

命中结果：
- **《地府货币膨胀：东亚父母该烧多少钱才能保证孩子不会乱花》**

这说明：
**一旦发现站内官方搜索接口，就优先用站内搜索，不要继续靠外部搜索引擎硬猜。**

---

### Step 5：拿单篇详情，确认文件地址
命中结果后，不要马上结束。

继续做：
- 请求单篇详情接口
- 确认 `pdf_url` / `file_url`
- 核对标题、作者、评分、分区

本次任务中使用：
- `/api/articles/{article_id}`

拿到了：
- 标题
- 作者
- 分类
- 分区
- 评分
- PDF 直链

---

### Step 6：下载到本地，再转发给用户
如果用户直接打开外链有困难，或者平台对链接支持差，就不要只发 URL。

更可靠的交付方式是：
1. 下载到本地临时路径
2. 用平台支持的文件发送能力直接发给用户

本次任务中：
- 下载到本地：`/tmp/shit_currency_inflation.pdf`
- 再通过 QQ 文件标签发给用户

这一步极大提高了“用户真的拿到文件”的成功率。

---

## 四、本次任务的关键经验

### 经验 1：目标模糊时，先找系统，不先找结果
比起一开始就反复搜论文名，更有效的是先摸清：
- 网站结构
- 数据入口
- 搜索方式

### 经验 2：浏览器挂了，不代表任务卡死
浏览器不可用时，仍可用：
- `curl`
- `web_fetch`
- 前端 JS 分析
- OpenAPI 文档
- JSON API 请求

### 经验 3：站内搜索往往比全网搜索更准
尤其是：
- 标题很怪
- 搜索词很长
- 站点内容偏小众
- 公开网页 SEO 不强

### 经验 4：最终目标是“送达”而不是“定位”
如果用户看不到外链 PDF，真正完成任务的方法是：
- 直接下载
- 直接发送文件

### 经验 5：把过程蒸馏成可复用方法
一次找到了，不算真正完成；
能总结成“下次还找得到”的 workflow，才算沉淀成资产。

---

## 五、可复用的最小模板

以后遇到类似任务，建议按这个最小模板走：

1. 确认官网
2. 抓前端 bundle / HTML
3. 找 API 域名
4. 看 OpenAPI / Swagger
5. 先拉列表看分页
6. 用站内搜索精确命中
7. 请求单篇详情
8. 找文件地址
9. 下载到本地
10. 直接发给用户

---

## 六、为什么这套方法值得写进项目

因为它体现的不是“会搜网页”，而是：
- 会识别网站结构
- 会在工具失效时换路线
- 会从前端反推后端接口
- 会把查找结果真正交付给用户
- 会把一次成功操作沉淀成可复用 workflow

这正是 OpenClaw 在真实工作流里的价值之一。
