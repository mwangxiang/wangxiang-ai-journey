# Workflow：如何用 OpenClaw 持续运维一个公开 GitHub 项目

这个 workflow 记录的不是“AI 帮忙写一篇文档”，而是更进一步的事情：

> 把 OpenClaw 作为长期协作对象，用来持续维护一个真实公开的 GitHub 项目。

这也是 `WangXiang AI Journey` 当前最重要的实践方向之一。

## 一、这个 workflow 解决什么问题

很多人会用 AI 做一次性任务，比如：
- 写一段文案
- 改一份文档
- 总结一次聊天

但公开项目的维护不是一次性任务，而是长期任务：
- 仓库结构要维护
- README 要持续更新
- 案例要补
- workflow 要沉淀
- 变更要记录
- GitHub 要持续推送和整理

所以这里要解决的问题是：

**如何让 OpenClaw 从“临时工具”变成“长期 GitHub 运维协作者”。**

---

## 二、适用场景

这个 workflow 适合：
- 个人公开项目
- build in public 项目
- AI 助手实践仓库
- 作品集型 GitHub 项目
- 主要由个人驱动、但希望持续更新的仓库

不太适合：
- 纯代码大团队仓库
- 严格企业审批流项目
- 不允许 AI 参与维护的私有项目

---

## 三、核心分工

### 人负责什么
人主要负责：
- 定方向
- 做最终判断
- 设定哪些内容值得公开
- 决定项目想呈现什么形象

### OpenClaw 负责什么
OpenClaw 主要负责：
- 维护仓库结构
- 撰写和修改文档
- 整理案例
- 沉淀 workflow
- 更新 CHANGELOG
- 执行 Git 提交与推送
- 维护 README 展示效果
- 按约定持续推进项目内容

这套分工的关键，不是“完全放权”，而是：

**人主导方向，AI 负责持续施工。**

---

## 四、最小可行维护循环

### Step 1：人提出方向或目标
例如：
- “把这个项目做成简历的一部分”
- “把最近这次折腾写成案例”
- “补几个成型 workflow”
- “优化首页，更像作品集”

### Step 2：OpenClaw 拆成具体维护动作
例如拆成：
- 改 README
- 新增 case study
- 新增 workflow 文档
- 更新 CHANGELOG
- 推送 GitHub

### Step 3：OpenClaw 执行 GitHub 运维动作
包括：
- 写文件
- 改文件
- 建目录
- 提交 commit
- push 到远端仓库

### Step 4：人做反馈
例如：
- 太空了
- 不够像作品集
- 再补几篇
- 这个方向可以继续

### Step 5：OpenClaw 根据反馈继续迭代
这样就形成一个长期循环，而不是一次性对话。

---

## 五、执行时的关键原则

### 1. 先结构，后堆内容
仓库先搭骨架，再逐步填内容。
否则越写越散，后面维护成本会越来越高。

### 2. 每次迭代都尽量落地到 GitHub
不是只在聊天里说“应该怎么改”，而是要真正改文件、提交、推送。

### 3. 每次新增内容都尽量可归档
新增内容优先归到：
- learning log
- case study
- workflow
- guide

这样长期看不会乱。

### 4. 公共展示内容必须脱敏
OpenClaw 可以帮助整理公开内容，但不代表可以把原始记录直接发出去。
任何公开前的内容都要经过脱敏检查。

### 5. 把 GitHub 运维本身也当成项目成果
不是只展示仓库内容，还要展示：
- AI 如何帮助运维
- 人机协作如何形成稳定工作流
- AI 如何替代一部分日常仓库维护工作

这本身就是有展示价值的项目能力。
