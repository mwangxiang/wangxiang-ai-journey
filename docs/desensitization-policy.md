# 脱敏公开规范（Desensitization Policy）

这个项目希望尽量公开真实过程，但公开不等于把一切原样贴出来。

目标是：
- 保留真实细节
- 保留过程感
- 保留学习价值
- 去掉足以造成隐私、安全、账号风险的内容

## 一、可以公开的内容

通常可以公开：

- 方法论
- 工作流设计
- 目录结构
- 工具选择思路
- 踩坑记录
- 错误现象
- 复盘结论
- 非敏感截图
- 已去除身份信息的对话片段
- 与公开项目直接相关的经验总结

## 二、必须去掉的内容

以下内容原则上不得公开：

- API Key / Token / 密码 / Cookie / Session
- 私人聊天中不适合公开的内容
- 手机号、邮箱、住址、证件、账号标识等个人信息
- 服务器 IP、SSH 入口、内网结构、端口暴露信息
- 付款、订单、财务、商业机密
- 任何能让别人直接获取权限的内容

## 三、推荐处理方式

### 1. 直接替换
把敏感值替换为：
- `[REDACTED]`
- `[HIDDEN]`
- `[PRIVATE]`

### 2. 泛化表达
例如：
- 把真实 IP 改成“某云服务器”
- 把真实 API 地址改成“某第三方服务”
- 把真实账号名改成“某账号”

### 3. 改写而不是照搬
对聊天记录、操作日志、失败记录，不一定要逐字公开。
在不损失核心信息的前提下，可以改写成：
- 问题背景
- 采取动作
- 结果
- 结论

## 四、发布前检查清单

每次准备公开内容前，至少检查：

- [ ] 有没有密钥、token、cookie
- [ ] 有没有手机号、邮箱、私人身份信息
- [ ] 有没有服务器/IP/内网等敏感基础设施信息
- [ ] 有没有不适合公开的原始聊天内容
- [ ] 有没有会让别人误解为可直接复现攻击路径的描述

## 五、处理原则

当你拿不准某个内容能不能公开时：

**宁可少公开一点，也不要为了“真实”把敏感信息带出去。**

这个项目要展示的是：
- 诚意
- 细节
- 执行力
- 学习能力

不是展示秘密本身。
